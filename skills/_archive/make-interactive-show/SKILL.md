# /make-interactive-show

从原始故事文件或已有预览项目制作完整的互动影游剧本。

## 核心模型：爽点驱动短剧集

每集约 3 分钟，遵循抉择驱动节奏：

```
[铺垫/蓄压 setup ~1.5min] → [抉择时刻 CHOICE] → [分支结果 resolution ~1.5min per option]
```

- **铺垫/蓄压**：建立压力、困境、挑衅，蓄积情绪张力，把主角逼到必须行动的墙角
- **抉择时刻**：在最紧张的一刻，画面定格——观众替主角选择行动方式
- **分支结果**：每个选项各自展开不同的爽点释放方式，并以钩子收尾引向下集

**一集 = 一个蓄压 + 一个抉择 + 两种爽法。**

铁律：选择 = 主角的行动决策，不是剧情走向。
- 观众控制的是"主角怎么做"，不是"接下来发生什么"
- 选择出现在主角被逼到墙角、必须立刻行动的那个瞬间
- 每个选项都是一种合理的、观众"如果是我也想试试"的行动
- 选项之后的内容是该行动的结果——不同的爽点释放方式

## 题材分类与题材爽点

### 代入感核心原则

**女频第一法则：观众必须能代入自己，爽得到位。**

- **铺垫 = 让观众替女主委屈**（"这也太欺负人了吧"）
- **爽点 = 让观众替女主出气**（"干得漂亮！痛快！"）
- **选择 = "此刻我该怎么做"**（选择出现在被逼到墙角的那一刻，选项是观众真正想尝试的行动）
- **冲突性 > 逻辑性，情绪性 > 故事性**（先抓住情绪，逻辑其次）

代入感设计要点：
1. 铺垫中的"虐"必须具体——谁在欺负、怎么欺负、为什么不公，越具体越有代入感
2. 爽点必须彻底——不半途而废，主角反击要直接、到位、不留余地
3. 旁观者反应是放大器——围观者的震惊/哑口无言 = 观众的爽感翻倍
4. 男主是NPC，女主才是玩家——女主的成就不依赖男性角色

### 题材自动检测

MINE 阶段第一步：根据故事内容自动识别题材，使用对应的爽点/钩子类型表。

| 检测关键词 | 题材 | 标签 |
|-----------|------|------|
| 穿越/重生/替嫁/空间/流放/宅斗/宫斗/嫡庶/世子/王妃/侯府 | 大女主古言 | `heroine_period` |
| 修仙/修炼/灵根/丹药/飞升/宗门/筑基/金丹/元婴 | 修仙 | `cultivation` |
| 总裁/豪门/商战/职场/霸道/首席/少爷/千金 | 都市 | `urban` |
| 末日/丧尸/异能/废土/避难所/基地 | 末世 | `apocalypse` |
| 悬疑/推理/密室/凶手/线索/真相 | 悬疑 | `mystery` |

未匹配时使用通用爽点表。

### 题材：大女主古言 (`heroine_period`)

> **核心爽感**：从被轻视到碾压全场的情绪翻转。观众在铺垫中感到"替女主委屈"，在爽点中感到"替女主出气"。
>
> **情绪公式**：被欺→反击→全场震惊→观众痛快

#### 大女主古言 · 爽点类型

| 类型 | 标签 | 描述 | 必要铺垫 | 代入感触发 |
|------|------|------|----------|-----------|
| 怼人打脸 | `verbal_slap` | 一句话噎死对方，让嘲讽者当场哑火 | 先有人出言侮辱/轻视/道德绑架 | "这话我也想说！" |
| 零元购/搬空 | `loot_sweep` | 用金手指把对方家底搬空，一毛不留 | 先展示对方的贪婪/不义/忘恩负义 | "搬！全搬走！痛快！" |
| 揭马甲 | `identity_reveal` | 隐藏的身份/能力当众曝光，震惊所有人 | 先被低估/嘲笑/当成废物 | "看看你们瞧不起的人是谁" |
| 护短 | `fierce_protection` | 挡在家人/弱者面前，一夫当关 | 先有人欺负弱者/欺负我的人 | "我罩的人你敢碰？" |
| 断亲宣言 | `cutting_ties` | 当众与渣家族/渣关系断绝 | 先被亲人抛弃/利用/当工具 | "不是你不要我，是我不要你" |
| 虐渣 | `punish_villain` | 让坏人付出代价，以彼之道还施彼身 | 先有人做了具体的恶事/伤害了具体的人 | "活该！报应！" |
| 白莲花翻车 | `expose_schemer` | 当众撕下伪善者的面具 | 先有人装无辜/暗中使坏/博同情 | "我早看穿你了" |
| 囤货炫富 | `resource_flex` | 别人饿着时我吃香喝辣，别人冻着时我穿皮裘 | 先展示所有人的困境/匮乏 | "羡慕吧？活该！" |
| 降维打击 | `dimensional_crush` | 用穿越/异能/现代知识碾压古代困局 | 先展示古代困境看似无解 | "小意思" |
| 男主折服 | `male_lead_awe` | 强势男主被女主的能力/魄力震撼 | 先展示男主的强大/傲慢/不屑 | "服不服？" |

#### 大女主古言 · 钩子类型

| 类型 | 标签 | 描述 | 代入感效果 |
|------|------|------|-----------|
| 暗中注视 | `secret_observer` | 神秘人（通常是男主）在暗处观察女主一切 | "谁在看我？什么来头？" |
| 身份猜疑 | `identity_suspicion` | 有人开始怀疑女主的真实身份/能力来源 | "要暴露了吗？怎么圆？" |
| 阴谋逼近 | `approaching_scheme` | 敌人的下一个阴谋即将发动，女主可能还不知道 | "小心！快想办法！" |
| 情感试探 | `emotional_probe` | 男女主互相试探，暧昧中带着警惕 | "他到底什么意思？" |
| 真相一角 | `partial_truth` | 揭开一角谜底，但更大的秘密浮出水面 | "原来如此…等等还有？" |
| 危机预兆 | `crisis_omen` | 更大的威胁/阴谋/势力即将降临 | "这次麻烦大了…" |

#### 大女主古言 · 铺垫升级模板

铺垫的"虐"必须层层递进，每次比上一次更过分，才能让爽点更有爆发力：

```
Lv.1  言语轻视    → "不过是个私生女/替嫁的"
Lv.2  行动排挤    → 不给食物/不让坐马车/故意刁难
Lv.3  公开羞辱    → 当众撕衣/砸砖/泼脏水
Lv.4  肉体威胁    → 鞭打/刀锋/追杀
Lv.5  灭绝式打击  → 斩草除根/陷害入狱/满门抄斩
```

每升一级，对应的爽点反击也必须升级：从嘴上反击 → 武力反击 → 势力碾压 → 掀翻棋盘。

## 参数

解析用户指令中的以下参数：

| 参数 | 必填 | 默认值 | 示例 |
|------|------|--------|------|
| `--file` | 是 | - | `/path/to/story.txt` 或已有项目目录 |
| `--episodes` | 否 | `50` | `30`, `50`, `80` |
| `--episode-duration` | 否 | `3m` | `2m`, `5m` |
| `--playthrough-ratio` | 否 | `0.6` | `0.5`, `0.7` |
| `--options-per-selection` | 否 | `2-3` | `2-4` |
| `--branching` | 否 | `heavy` | `light`, `medium`, `heavy` |
| `--note` | 否 | `""` | `"修仙+种田+轻松幽默"` |
| `--lang` | 否 | `zh` | `en`, `zh` |
| `--output` | 否 | `~/Downloads` | `/path/to/output` |
| `--chapter-range` | 否 | 全部 | `1-80` |

如有必填参数缺失，向用户询问。

**语言规则**: 所有输出内容（剧本、故事圣经、结构描述、评估报告、PDF/DOCX）一律使用中文，除非情节或台词需要外语。JSON key 保持英文，value 使用 `--lang` 语言。

**参数推导**（自动计算）：
```
total_duration = episodes * episode_duration
playthrough_episodes = round(episodes * playthrough_ratio)
playthrough_duration = playthrough_episodes * episode_duration
```

**智能输入检测**：
- 含 `state.json`（mode=preview）→ **预览续接**：跳过 CHUNK/DISTILL/BIBLE/MINE/TOPOLOGY，直接进入脚本
- 含 `beats.json` → 跳过 MINE
- 含 `structure.json`（有完整拓扑）→ 跳过 TOPOLOGY
- 含 `chunks/manifest.json` → 跳过 CHUNK
- 含 `story_bible.json` → 跳过 DISTILL + BIBLE

## 制作流程总览

```
Phase 1:  CHUNK      → 切分大文本（可复用）
Phase 2:  DISTILL    → 提炼摘要（并行）
Phase 3:  BIBLE      → 合并故事圣经
Phase 4:  MINE       → 爽点挖掘 ★
Phase 5:  TOPOLOGY   → 集级DAG拓扑（可从预览复用）
Phase 6:  EPISODES   → 并行设计每集场景 ★★
Phase 7:  SCRIPT     → 并行编写每集脚本 ★★
Phase 8:  VERIFY     → 一致性校验（3轮）
Phase 9:  EVAL       → 质量评估
Phase 10: REVISE     → 修改（如需）
Phase 11: RENDER     → 生成 PDF + DOCX + JSON
```

★★ = 主要并行化步骤

## 执行指令

### Step 0: 初始化与智能检测

```bash
PROJECT_DIR="$OUTPUT_DIR/interactive_show_$(date +%Y%m%d_%H%M%S)"
mkdir -p "$PROJECT_DIR"
```

检测输入类型，决定跳过哪些步骤。保存参数到 `$PROJECT_DIR/args.json`。

### Step 1-3: CHUNK → DISTILL → BIBLE

与 `/make-interactive-show-preview` 完全相同。从预览项目续接时全部跳过。

### Step 4: MINE 爽点挖掘

**4a. 题材检测**：读取 story_bible 和原文摘要，根据「题材自动检测」表匹配题材。将检测结果写入 `$PROJECT_DIR/args.json` 的 `detected_genre` 字段。

**4b. 爽点挖掘**：使用检测到的题材对应的爽点/钩子类型表进行挖掘。每个 beat 必须标注：
- `satisfaction_type`: 使用题材专属标签（如 `verbal_slap`、`loot_sweep`）
- `hook_type`: 使用题材专属标签（如 `secret_observer`、`identity_suspicion`）
- `immersion_trigger`: 代入感触发语（如 "这话我也想说！"），帮助脚本编写时把握情绪

**4c. 代入感校验**：每个 beat 自检——"观众能否在铺垫中感到替主角委屈？爽点中感到替主角出气？" 不通过的 beat 重新设计铺垫。

与 `/make-interactive-show-preview` Step 4 的基础流程相同。从预览项目续接时跳过。

### Step 5: TOPOLOGY 集级DAG

与 `/make-interactive-show-preview` Step 5 完全相同。从预览项目续接时直接读取已有 `structure.json`。

#### 分支复杂度表

| `--branching` | 总集数 | 最少3路 | 最少2路 | 最少总分叉 |
|---------------|--------|---------|---------|----------|
| heavy | 31-50 | 2 | 6 | 8 |
| medium | 31-50 | 1 | 4 | 5 |
| light | 31-50 | 1 | 2 | 3 |

#### structure.json 精简结构

与预览技能完全相同的精简格式。structure.json 只负责 **DAG 拓扑 + 选择设计**：

- 一个 `episodes` 数组即是图——`choice.options[].next` 定义边
- 不单独维护 `fork_points`（从 episodes 中 choice.type="fork" 的集可推导）
- 每集只保留：id、title、thread、chapter_source、summary（≤40字）、beat_type、beat_label、choice
- 角色/世界观在 story_bible.json，爽点详情在 beats.json

### Step 6: EPISODES 并行设计每集场景 ★★

**为每个集启动一个 Task 代理，全部并行。**

每个代理接收：
1. `story_bible.json`（角色/世界观）
2. `structure.json` 的 `episodes` 数组（DAG 全图）
3. `beats.json` 中本集对应的 beat 详情（setup/satisfaction/hook）
4. 本集的 `choice.type`（fork 或 flavor）
5. 前驱/后继集信息
6. `--note`、`--lang`

#### 代理输出格式

每集固定结构（遵循抉择驱动节奏）：

```json
{

  "scenes": [

      "id": "EP01-C1",

      "type": "escalation",
      "description": "升级内容：矛盾激化，主角被逼到墙角..."
    },
    {
      "id": "EP01-C3",
      "name": "抉择时刻场景名",
      "type": "choice_moment",
      "description": "主角必须立刻行动的那一刻..."
    },
    {
      "id": "EP01-C4A",
      "name": "选项A结果场景名",
      "type": "resolution_a",
      "satisfaction_type": "hidden_power",
      "description": "选项A的爽点释放 + 钩子收尾..."
    },
    {
      "id": "EP01-C4B",
      "name": "选项B结果场景名",
      "type": "resolution_b",
      "satisfaction_type": "outwit",

    }
  ],
  "choice": {
    "question": "主角此刻面对的困境——必须行动的那一刻",
```

#### 代理提示模板

```
你是一位互动短剧设计师。请为以下集设计场景和选择。

## 铁律：选择 = 主角此刻的行动
选择必须出现在主角被逼到必须行动的那个瞬间。
- ❌ 事件演完了再问"下一步怎么办"——这是假选择
- ✅ 事件正在发生时问"此刻你怎么做"——这才是真选择
- 每个选项是主角的不同行动方式（暴力/智取/隐忍/冒险...）
- 选项之后的内容 = 该行动的不同结果（不同的爽点释放方式）

## 本集爽点+钩子（来自挖掘阶段）
- 爽点类型：{satisfaction_type} ({satisfaction_label})
- 铺垫概要：{beat.setup}
- 爽点概要：{beat.satisfaction}
- 钩子类型：{hook_type} ({hook_label})
- 钩子概要：{beat.hook}

## 本集信息
- ID: {episode_id}，标题: {title}
- 时长: 3分钟，路线: {thread}
- 前驱集: {predecessors}，后继集: {successors}

## 设计要求
- 前半场（~1.5min）：铺垫蓄压 + 危机升温，把主角逼到必须行动的临界点
- 抉择时刻：在最紧张的那一刻，主角必须做出行动选择
- 后半场（~1.5min per option）：每个选项展开不同的爽点释放 + 钩子收尾
- question = 主角此刻面对的困境，选项 = 主角的不同行动方式
- **铁律：每一集都必须有观众选择**（不存在无互动的"过场集"，这是互动剧不是电视剧）
- **铁律：每个选项的 next 必须指向不同的集**（没有例外）
- 部分选项可指向死胡同集（错误选择 = 游戏结束）
- 选项 `text` ≤ 8 字，唯一，不得重复
- 每个选项的 outcome 必须不同
- question 必须紧扣本集蓄压高潮，不得使用通用问题
- 如果本集是死胡同集（is_dead_end=true），choice 为 null

## 故事圣经
{story_bible_summary}

## 创作方向
{note}

请输出 JSON 格式。
```

### Step 7: SCRIPT 并行编写脚本 ★★

为每集生成完整脚本。**全部并行执行。**

每集脚本遵循专业影视剧本格式（参照《天坑追匪》剧本规范）。

#### 场景头格式

每个场景以两行表头开始：

```
场：001    景：青牛镇·二狗子家菜地
时：日外    人：韩二狗、翠花嫂、张老汉
```

- `场` = 全剧连续场号（三位数字 001, 002, ...），跨集连续编号
- `景` = 场景地点名
- `时` = 时间环境（日内、日外、夜内、夜外、黄昏、清晨等）
- `人` = 本场出场角色列表

#### 舞台指示

以 `▲` 开头，用自然影视语言描述画面。镜头提示以括号嵌入，**不使用固定 【镜头】 标签**：

```
▲篝火旁大腿帮子擦枪，已经搭好的两个帐篷，夜色渐深。
▲（特写）韩二狗手指微微颤抖，捏着那株通体碧绿的灵葫。
▲（手持跟拍）四人在迷雾中穿行。
▲（主观）赵工迅速进兰子帐篷。
▲（前景火，后景两人，变焦到火）
▲（从脚起逼近）正摇摇晃晃走近。
```

常用镜头提示词：特写、近景、中景、全景、广角、俯拍、仰拍、推、拉、摇、跟、手持、主观、变客观、快切、慢动作、定格、渐暗

#### 台词格式

```
角色名：台词内容
角色名（表情/动作）：台词内容
角色名：（外语标注）台词内容
角色名（顿了顿）：……台词内容
角色名：（喃喃自语）内心想法……
```

内心独白不使用专门标记，通过表演指示自然呈现：`（喃喃自语）`、`（心想）`、`（暗忖）`、`（自言自语）`、`（喃喃）`。

单独一行，用于提示 AI 生成画面：

```

```

#### 选择格式

选择以 `选择 NNN` 开头（NNN = 当前场号），**紧接下一行必须是 `问题：` 行**，然后是选项：

```
选择 014
问题：暗处传来脚步声，篝火旁和帐篷里都可能有危险——
014A：去兰子帐篷
014B：去篝火旁
```

- `问题：` 行描述主角此刻面对的困境，让玩家知道自己在为什么做选择
- 选项文本 ≤ 8 个中文字符（简短有力）。好示例："先保护灵葫"、"救文物箱"、"拼死一搏"
- **铁律：每个 `选择 NNN` 之后必须有 `问题：` 行，没有例外**

#### 分支内容（分节格式）

不同选项的内容以清晰分节标题书写，每个分支以 `━━━` 分节线标注：

```
━━━━━━━━━━ 014A：去兰子帐篷 ━━━━━━━━━━

▲赵工走向兰子的帐篷。
...
选择 014A 结束，接第一章 018 场

━━━━━━━━━━ 014B：去篝火旁 ━━━━━━━━━━

▲赵工向大腿帮子篝火旁走去。
...
选择 014B 结束，接第一章 019 场
```

三路分支同理，分为三节。**禁止使用 `--- NNNA：xxx ---` 并列列格式**（难以阅读和渲染）。

**Flavor 选择脚本规则**（当 choice_type == flavor 时）：
- 虽然所有选项都通往同一集，但选择场景本身必须有差异化内容
- 每个选项分支应有 ≥ 100 字的不同内容，展现不同的角色处理方式/风格
- 差异化体现在具体的行动、对话、氛围，而非仅改一两个词

**脚本要求**:
- 每集节奏遵循 铺垫蓄压→抉择时刻→分支结果 结构
- 选择出现在集的中间（蓄压到顶的时刻），不是末尾
- 选择之前（~1.5min）：铺垫蓄压 + 危机升温，主角被逼到必须行动
- 选择之后（~1.5min per option）：每个选项的结果场景，含该选项的爽点释放和钩子
- 铁律：选择 = 主角此刻的行动，不是事后"下一步怎么办"
- **铁律：每个 `选择 NNN` 之后必须有 `问题：` 行**（描述主角此刻困境，让玩家理解选择背景）
- **铁律：分支内容使用 `━━━━━━━━━━ NNNA：xxx ━━━━━━━━━━` 分节格式**，禁止并列列格式
- **铁律：选项文本不得包含模板标记**（如 `（≤8字）`、`（选项）` 等）
- 铺垫要有"先抑"的压力感（被嘲笑/被围困/被欺压）
- 分支结果要有爆发力——每个选项是不同的爽法
- 每个分支结果的末尾以钩子收尾（为下一集蓄力）
- 舞台指示生动有电影感，用自然镜头语言（括号嵌入提示词）
- 台词自然符合角色身份
- 每集脚本 800-1500 字（3分钟节奏）
- 死亡结局用 `BE：描述` 标注

### Step 8: VERIFY 一致性校验

**关键步骤。脚本是源真相。**

6 轮校验，结果写入 `$PROJECT_DIR/verification_log.md`。

**第 1 轮：脚本 → 结构一致性**
- 每集脚本的选择与 structure.json 匹配
- 跳转目标（choice.options[].next）匹配
- 爽点类型（beat_type）匹配
- 选项文本 ≤ 8 个中文字符（超出的标记为错误并修复）
- 不一致 → 更新 structure.json

**第 2 轮：脚本 → 故事圣经一致性**
- 角色名、世界观、情节一致
- 不一致 → 更新 story_bible.json

**第 3 轮：因果一致性**
- 角色死亡追踪：已死亡角色不得在死亡集之后出场
- 能力/物品时间线：不得在引入前使用
- 新角色引入：首次出场集必须有介绍场景
- 不一致 → 修复脚本

**第 4 轮：格式合规**
- 每场以标准两行表头开头（`场：NNN 景：xxx` / `时：xxx 人：xxx`）
- `▲` 舞台指示使用自然镜头语言，镜头提示以括号嵌入，不使用固定 【镜头】 标签
- 选择格式为 `选择 NNN` + `问题：xxx` + `NNNA：选项文本`，选项文本 ≤ 8 个中文字符
- **铁律：每个 `选择 NNN` 之后必须紧跟 `问题：` 行**（描述主角此刻面对的困境）
- 分支内容使用 `━━━━━━━━━━ NNNA：xxx ━━━━━━━━━━` 分节格式，禁止 `--- NNNA：xxx ---` 并列列格式
- 选项文本不得包含模板标记（如 `（≤8字）`）
- 场景过渡有明确跳转标注（`NNN 场结束，接 NNN 场`）
- 不合规 → 修复脚本

**第 5 轮：选择质量校验**
- 每个选择有 ≥ 2 个选项
- 同一选择内无重复选项 text
- 同一选择内无重复选项 outcome
- question 不使用通用模板（"接下来怎么办"、"你的选择是"等）
- choice.type 与 options[].next 一致（fork 的 next 不同，flavor 的 next 相同）
- 选择是否出现在脚本的蓄压高潮处（检查选择位置在脚本中间，而非最后10%）
- 选项是否描述主角的行动方式（而非"下一步计划"类问题）
- 选择之后每个选项是否有 ≥ 3 行分支结果内容（不是只有一行跳转）
- 违反 → 重写该集的选择

**第 6 轮：交叉校验**
- structure.json × story_bible.json 内部一致
- beats.json 与最终集分配一致

### Step 9: EVAL 质量评估

读取 `$SKILL_DIR/eval_criteria.md`。评估重点：

1. **爽点质量**：每集爽点是否有足够铺垫？释放是否有力？
2. **钩子质量**：每集钩子是否制造了"不看下一集会死"的冲动？
3. **节奏一致性**：铺垫蓄压→抉择时刻→分支结果 节奏是否严格遵循？
4. **分支差异性**：不同路线的爽点是否真正不同？
5. **选择质量**：选择是否出现在蓄压高潮处？选项是否为主角的行动方式？

写入 `$PROJECT_DIR/eval_report.md`。
