# CREATIVE_WRITING_SKELETON.md — 骨架生成指令

你输出的是**结构化 JSON**，harness 做确定性校验，违规即拒绝。
此阶段生成**骨架**——节点的 `skeleton` 节拍数组。**skeleton 是本节点剧情的唯一载体**，包含所有关键事件、角色互动、对白要点（没有单独的 summary 字段）。
后续 prose 阶段只依据 skeleton 做文学润色（镜头语言、气氛描写、台词扩写），不新增剧情。

---

## 输入

- **端点 A、B**（外侧接口冻结）
- **故事圣经**（世界观、人物、背景）
- **章节范围** + 原文文本
- **未放置高光**（带权重）— 尽量覆盖高权重高光，放入 `covers`
- **目标**：`entryA_state`、`exitB_contract`、`invariants`、`varying_DO_NOT_REFERENCE`
- **注册表**（Registry）— 所有已声明事实的 ID、初始值、不变量标记
- **扩展类型**：
  - `LENGTH_EXTENDING` — 替换 A→B 直连边
  - `BRANCH_ADDING` — 保留 A→B，旁添平行路线

---

## 输出：每个节点必须包含的字段

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | string | 节点唯一标识 |
| `kind` | string | `prologue` / `scene` / `bottleneck` / `ending` |
| `skeleton` | ContentElement[] | **本节点剧情的唯一载体**：覆盖全部核心剧情节拍（见下方要求），不写审美性铺陈 |
| `planned_duration_min` | number | Phase 4 扩写后的目标拍摄/阅读时长（分钟） |
| `chapters` | [int, int] | 覆盖的原作章节范围 |
| `covers` | string[] | 覆盖的高光 ID |
| `produces` | Effect[] | 本节点建立的事实 |
| `requires` | Req[] | 进入前必须已持有的事实 |
| `entry_invariants` | Req[] | 进入时必须成立的不变量 |
| `ending` | string | `NONE` / `ENDING` / `DEAD_END` |
| `question` | string\|null | ≤30 中文字（结局为 null） |
| `choices` | Choice[] | 非结局必须恰好 2 个。两个选择可指向**同一目标**（此时各自的 `state_delta` 必须不同——选择写状态而非分叉）；否则目标必须不同 |
| `entry_context` | string | WHERE/WHEN 进入场景 |
| `exit_context` | string | WHERE/WHEN 离开场景 |

### skeleton 要求（核心！）

skeleton 是本节点的**唯一剧情载体**。prose 阶段依据 skeleton 节拍润色，不会发明新剧情。
因此 skeleton 节拍合起来必须覆盖所有关键信息：

1. **出场角色**：本节点出场的所有角色都要在节拍里点名（scene_header 的 characters + 动作/对白）
2. **关键事件**：按时间顺序写成节拍（3-8 个事件节拍，用 action）
3. **核心对白要点**：关键对话写成 dialogue 节拍——谁对谁说了什么意思（不需完整台词）
4. **物品/证据**：关键物品的获取、使用或展示必须有对应节拍
5. **场景变化**：场景转换用新的 scene_header 节拍
6. **情感转折**：角色态度或关系的变化要落到某个节拍上

**节拍要覆盖的内容示例**（下面是这些节拍合起来必须讲清的剧情，不是输出格式）：
```
出场：主角、敌对者、潜入者（伪装成随行人员）。
事件：(1) 主角在营地发现敌对者偷偷在共用物资里做手脚。(2) 她不动声色地替换掉被动过的物资。(3) 潜入者走来搭话，问她为何深夜未睡，语气试探。(4) 主角以借口搪塞，但注意到此人目光锐利，身份可疑。(5) 敌对者发现物资被换，愤怒质问主角。
对白要点：潜入者试探「姑娘胆子不小，敢一个人在外面走？」 / 主角反问「你一个随行的，倒是对这队人格外上心」 / 敌对者威胁「你最好别多管闲事」
转折：主角开始怀疑「随行人员」的真实身份
```

**禁忌**：
- ❌ 模糊概括：「主角与敌人对峙」 → 谁是敌人？对峙什么？
- ❌ 遗漏角色：「展示了铁证翻案」 → 什么铁证？从哪来的？
- ❌ 跳跃逻辑：「最终获胜」 → 怎么获胜的？

### skeleton 格式要求（薄、结构化）

skeleton 是**核心剧情执行稿**，不是最终 prose。

- 必须使用结构化 ContentElement 数组（scene_header / action / dialogue / narration）
- 第一个节拍必须是 `scene_header`
- 只写核心剧情动作、关键对白意思、关键物品/证据
- 不写长气氛描写、不写修辞铺陈、不扩写成完整剧本
- 非 DEAD_END 节点 ≥5 个节拍，DEAD_END ≥3 个；ENDING/DEAD_END 也必须覆盖核心结局动作
- skeleton 必须与 produces、question、choice resolution 一致
- planned_duration_min 必须与剧情重量匹配：非 DEAD_END 节点 2.0-5.0 分钟；DEAD_END 节点 1.0-1.5 分钟
- 不要输出 `content` 字段——prose 在 Phase 4 从 skeleton 生成

**汇合节点特别要求**：

1. **世界状态兼容性**（最重要）：两条路径只有在**世界状态兼容**时才能汇合到同一个节点。如果路径A中反派已被擒获，路径B中反派仍然自由行动，这两条路径**绝对不能**汇合。如果路径A中角色甲已经建立信任，路径B中角色甲还是陌生人，也**不能**汇合。遇到世界状态不兼容时，必须给每条路径设置**独立的后续节点**。

2. **内容路径中立性**：skeleton 节拍中的每个事件和角色引用必须对所有路径的读者都讲得通。不能引用只在某一条路径上发生的事件或获得的物品。

3. **只出现共同角色**：汇合节点的 skeleton 只能包含所有路径都保证会遇到的角色。如果某个角色只在部分路径出场，不能在汇合节点中提到她/他。

### state_delta（选择写状态）

- 每个 Choice 可携带 `state_delta`：选择后追加的状态变化，如 `[{"fact": "player.bond_with_x", "value": true}]`
- **同目标双选择**：两个选择指向同一目标节点时，各自的 `state_delta` 必须不同——玩家的选择留下不同的痕迹（性格倾向、关系变化），剧情主线汇合但状态分流
- state_delta 中的 player.* 事实是运行时玩家状态：**不要**在后续节点的 `requires`/`label_requires` 中引用它们（汇合后这些事实是 VARIES，引用会触发 D4）
- 新事实照常在 `new_facts` 中声明

## 会被拒绝的情况（D 检查）

- **D1**: `requires` 中的事实在所有进入路径上的 guaranteed 值不匹配
- **D2**: `label_requires` 在选择时刻的状态不满足
- **D3**: `entry_invariants` 在进入时不成立
- **D5**: 使用了未在注册表中声明的事实 ID
- **D6**: 图有环
- **D7**: 翻转了不变量事实（invariant）
- **D9**: 结构违规 — 具体包括：
  - `kind` 不是 4 种之一
  - 缺少 `entry_context` 或 `exit_context`
  - 非结局节点不是恰好 2 个选择（同目标的两个选择必须有不同的 state_delta，否则视为纯装饰选择被拒绝）
  - 结局/死胡同有选择
  - `question` 超 30 字 / `label` 超 8 字
  - `question` 使用泛模板（如何应对/如何回应/怎么办/如何处理/怎么应对）
  - 两个选择 label 前两字相同（必须使用不同动词）
  - `resolution` 不是恰好 2 个短节拍
- **D10**: 不可达或无法到达 ENDING
- ~~D11 已退役~~：连续性看**剧情**不看地点——转场切换地点是正常电影语法

**注意**：骨架阶段不检查 prose 长度和格式（D9 prose 相关检查跳过；skeleton 只需满足节拍数下限）。

## 断言 vs. 预设 — 防 Bug 核心规则

对每个涉及的事实问：*如果读者第一次看到这个节点，能否讲得通？*

- **能 → 断言**：首次出现即引入。`requires: []`。人物直接出场是正常的。
- **不能 → 预设**：回引了之前展示过的内容。加入 `requires`。

**预设触发词**：确指表达（"*那把*铜钥匙"）、"又"、"还是"、"再次"、"如约"、"你还记得"、"同一个人"。
不确指首次出现（"一个叫玛拉的女人"）是断言，无需 requires。

---

## 事实规则

- 优先复用注册表中已有的 fact ID
- 新增事实必须在 `new_facts` 中声明
- 不可翻转 invariant 事实（初始值永远不变）
- `produces` / `requires` 保持最小化
- 事实 ID 前缀：`player.*`（玩家已知）、`char.<名>.* `（角色知道）、`world.*`（客观事实）

### 关键人物/物品引入必须声明

当一个节点**首次引入**一个有名有姓的角色或关键物品（后续节点的剧情依赖该角色/物品）时，**必须**在 `produces` 中声明事实：
- 引入角色：`produces: [{"fact": "char.<名>.introduced", "value": true}]`
- 获得物品：`produces: [{"fact": "player.has_<物品>", "value": true}]`

当一个节点引用前面节点引入的角色/物品时，**必须**在 `requires` 中声明：
- `requires: [{"fact": "char.<名>.introduced", "value": true}]`

**为什么？** 如果一个汇合节点（多条路径可达）的剧本提到"那个账房侄子"，但有一条路径根本没有遇到过这个人，读者会一头雾水。通过 `requires` 约束，系统可以确保所有路径都满足前置条件。

---

## 不变量事实详解

注册表中标记 `invariant=true` 的事实是**世界公理** — 值被锁定在 `initial`，永远不能改变。
你的 `produces` 不可将 invariant 事实设为不同于其初始值的值。
skeleton 节拍中也不可暗示 invariant 事实发生了变化。

---

## 生成顺序（MANDATORY）

你必须按此顺序思考和生成每个非结局节点：

1. **先定选择困境（竞争性收益 competing goods）**：基于 A 的 exit_context 和原文，
   先找出本场两个**都积极、玩家都想要**的收获 A 和 B（财富/真相/情报/立威/盟友/时机/
   保命…），A≠B 且同赌注层级。选项1=争取A（舍B），选项2=争取B（舍A）。
   - question 是"两样都想要却只能拿一样"的取舍，不是"如何应对/回应"、也不是"做 vs 不做"
   - 两个 label 必须各自争取一个**正向收获**，不得是另一项的否定/忍让/不做
   - **被支配检验**：如果某选项只能写出"避免了对方的坏处"而没有自己的正向收获，
     说明这不是真选择——重铸（问"忍/退买到了什么？"）或改为无选择的叙事节点
2. **再写 skeleton 节拍**：从 A 的 exit_context 出发，把角色、事件、对白要点、转折逐拍写出
3. **最后填元数据**：produces, requires, chapters, covers

---

## 选择规则

- `label` ≤8 中文字，写玩家动作，不写结果，不剧透目标内容
- **label 必须是具体动作**（对象+手段，如「贿银五十两」「割舌立威」），禁止抽象策略词
  （「以智取胜」「谨慎行事」一类的范畴词）。question 与两个 label 必须指同样的两个具体行动，
  用词一致——question 里的说法和 label 不能是两套词汇
- 每个选择通往**实质不同的状态或路径**
- 禁止伪选择、不合逻辑的选择、倒退选择
- `resolution` 恰好 2 个短句，展示选择结果
- 如果标签提及目标节点才揭示的事物 → 加入 `label_requires`

---

## 选择张力 — 让选择嵌入场景冲突

**核心原则**：选择不是菜单，是角色在压力下的抉择。skeleton 节拍必须把玩家推到不得不选的临界点。

**选择触发时机（强制）**：question 必须由本节点**最后 1-2 个骨架节拍**逼出——一个突发事件、反转或最后通牒把主角推到张力顶点，question 问的是"此刻如何应对眼前这件事"，绝不是脱离场景的泛策略问题。如果删掉最后两个节拍后 question 依然成立，说明选择没有挂在张力顶点上，必须重写。

**节拍职能标注（role 字段）**：每个骨架节拍标注其戏剧职能 `role: buildup|payoff|surprise|decision_trigger|recap|preparation|aftermath`。非结局节点的**最后一个节拍必须是 `decision_trigger`**（就是逼出 question 的那个事件）；爽点节拍标 `payoff`；汇合节点开场节拍标 `recap`。验证器据此机械检查"选择是否挂在张力顶点"。

### 五种张力模式（至少用一种）——但都必须满足"两个正向收获"底线

1. **竞争性收益（首选）**：两个选项各争取一个玩家都想要的好处，互为机会成本
   （财富vs真相、立威vs藏锋探敌、救人vs追凶）。选 A 即舍 B。
2. **风险不对称**：一个选项稳妥拿到 X，另一个高风险博更大的 Y（不是"博 vs 不博"）
3. **角色考验**：两个收获分别迎合角色不同的价值观（忠 vs 义、复仇 vs 自保），都诱人
4. **NPC 期望**：满足 NPC-甲 的代价是辜负 NPC-乙，两边各有所得
5. **不可逆性**：选择关上一扇门同时打开另一扇——两扇门后都有玩家想要的东西

**禁止**：approach-avoidance（"做大胆的事 vs 忍气吞声"）。任一选项若只是另一项的
否定或缺席，就不是合法选择点。

---

## 场景连续性 — 剧情连贯，地点自由

**核心原则**：连续性约束的是**剧情因果与时间**，不是地点坐标。

- 转场切换地点完全正常（"次日清晨·王府"），写出转场即可
- 时间只能前进（除非明确闪回，用 narration 标注）
- 在哪里发生戏，跟随**原著**：原文把这场戏放在哪里，你就放在哪里
- 不可接受的是：同一场戏中无法解释的瞬移、时间倒退、因果断裂

---

## 禁忌

- 不要预设 `entryA_state` 中没有的事实
- 不要翻转不变量
- 不要在有注册表 ID 的情况下发明新名字
- 不要在 skeleton 节拍中引用 `produces`/`requires` 中未声明的内容
- 不要在 skeleton 的 narration 节拍中交代主角身份/前史/能力来源（这些必须靠 action/dialogue + 他人反应演出）
- 不要引用 `varying_DO_NOT_REFERENCE` 中的事实
